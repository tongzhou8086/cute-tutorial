# 01 — JIT: "trace on every call" vs. explicit `cute.compile`

This experiment pins down a subtle piece of CuTe DSL's JIT model: **calling a
`@cute.jit` function re-traces your Python on _every_ call**, even when the
compiled kernel is served from cache. The only way to skip the trace is to
compile explicitly with `cute.compile` and call the returned executor.

## The two stages

CuTe DSL splits execution into two stages, and the distinction explains
everything below:

| Stage | When it runs | Runs on | Example in the code |
|-------|--------------|---------|---------------------|
| **meta-stage** | During Python tracing (compile time) | CPU / host | plain `print(...)`, `for`, `if` |
| **object-stage** | When the compiled kernel launches | GPU / device | `cute.printf(...)` |

Ordinary Python (`print`, loops, conditionals) executes on the host *while the
kernel is being traced* and is used to build the kernel. `cute.*` calls like
`cute.printf` are not executed during tracing — they are emitted as
instructions into the generated kernel and run later on the GPU.

## The surprising behavior (part 1 of `main.py`)

```python
import cutlass
import cutlass.cute as cute

@cute.jit
def add_dynamicexpr(b: cutlass.Float32):
    a = cutlass.Float32(2.0)
    result = a + b
    print("[meta-stage] result =", result)              # host, at trace time
    cute.printf("[object-stage] result = %f\n", result) # GPU, at launch time

add_dynamicexpr(5.0)
add_dynamicexpr(5.0)
```

Output:

```
[meta-stage] result = ?
[object-stage] result = 7.000000

[meta-stage] result = ?
[object-stage] result = 7.000000
```

Two things look surprising at first, and both are correct:

### 1. Why `[meta-stage] result = ?`

`b` is declared as `cutlass.Float32`, so the tracer treats it as a **dynamic
runtime value**. During tracing it is a *symbolic placeholder* — the compiler
knows its type (32-bit float) and which ops are legal, but not its value. So
`result = a + b` is a symbolic expression, and a plain `print` of it renders as
`?`.

The concrete `5.0` you passed is held aside and injected into the kernel only at
the object-stage, which is why `cute.printf` correctly shows `7.000000`.

### 2. Why the meta-stage prints on *every* call

This is the counter-intuitive part. In JAX / `torch.compile` / Numba, a second
call with a matching signature hits a cache and **skips your Python entirely**.
CuTe DSL does **not** work that way.

Per the [official JIT caching docs][cache-docs], the implicit cache key is the
hash of:

- the **MLIR bytecode** generated by CuTe DSL,
- all CuTe DSL Python source files,
- all CuTe DSL shared libraries,
- all CuTe DSL environment variables.

The cache is **not** keyed on the Python argument type signature. To compute the
MLIR hash, CuTe DSL has no choice but to **run your Python (the meta-stage) on
every call** to regenerate the MLIR, then hash it and look it up.

So the sequence for a direct `@cute.jit` call is always:

1. **Trace** — execute your Python to build the MLIR. *(your `print` fires here, every time)*
2. **Hash** — hash the resulting MLIR bytecode.
3. **Cache check** — look up the hash in the JIT-executor cache.
4. **Execute** — on a hit, skip the expensive MLIR→PTX compilation and reuse the cached binary; on a miss, compile and store it.

The implicit cache saves you from the slow compilation step (steps toward PTX),
but it **cannot** save you from the Python trace — so host-side side effects
like `print` run on every call. `no_cache=` (default `False`) only toggles
step 3/4; it does not change step 1.

> Why the cache must work this way: globals get baked into the MLIR as
> constants. In the docs' example, changing a module global `a = 1` → `a = 2`
> produces different MLIR with identical Python source, so a source-based key
> would be wrong. Re-tracing every call is what lets the cache notice that.

## The fix: explicit `cute.compile` (part 2 of `main.py`)

To get true "trace once, run many" behavior — no repeated meta-stage, no
per-call host tracing overhead — use CuTe DSL's **Zero Compile** feature.
`cute.compile` traces exactly once and returns a **JIT Executor** object:

```python
import cutlass
import cutlass.cute as cute

@cute.jit
def add_dynamicexpr(b: cutlass.Float32):
    a = cutlass.Float32(2.0)
    result = a + b
    print("[meta-stage] result =", result)
    cute.printf("[object-stage] result = %f\n", result)

# Trace ONCE here. The meta-stage print fires exactly once, now.
compiled_add = cute.compile(add_dynamicexpr, cutlass.Float32(0.0))

print("\n--- Compilation done. Running on GPU... ---\n")

# Call the executor, NOT the @cute.jit function. No tracing, no meta-stage.
compiled_add(5.0)
compiled_add(10.0)
```

Output:

```
[meta-stage] result = ?

--- Compilation done. Running on GPU... ---

[object-stage] result = 7.000000
[object-stage] result = 12.000000
```

The meta-stage runs **once**, during `cute.compile`. Each subsequent
`compiled_add(...)` call skips MLIR generation and Python tracing entirely: it
parses the Python runtime arguments, converts them to C-ABI–compatible types,
and launches the host function directly.

## Dump Generated IR

One way is to run the command with `CUTE_DSL_KEEP=ir` which dumps the generated IR to files in the current working dir.
